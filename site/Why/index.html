The story

Why SQLite.Framework was made

I want to start far back, because it helps explain my frustration with the SQLite tools for mobile that exist today.

When I started working, I was put on mobile Xamarin apps that needed a strong offline mode. My technical lead told me to use Entity Framework, because that is what we used on the backend.

There was a problem. To make the offline mode work, we had to store large amounts of data in the database. On the Nexus 5 phone we used for testing, that load took over 10 minutes with Entity Framework. Slow Entity Framework on mobile is a known problem that others have reported too, like this issue.

My lead did not want us to write raw SQL for queries or inserts. So we switched to sqlite-net-pcl. It worked well. The app had no complex queries, and we ran them in sync on a background thread. I was naive back then, and did not notice that most queries were read into memory first, then projected or joined in C# instead of in SQLite.

That did not change on the next few projects. Then a few years passed with no new mobile app. And at some point I realized:

If you know the framework and know your data, you need not fear the result of a hundred releases. Sun Tzu, The Art of Code

Then a new app changed everything. It had very complex queries. Administrators had to pull millions of rows (think of every road in a country). By then we had forgotten why we stopped using Entity Framework, so we tried it again. We found out that an administrator needed over 30 minutes to log in once all that data loaded. This time we decided to use custom inserts to fix the speed.

But life gave us another blocker we could not solve. On iOS, the code that Entity Framework builds at runtime broke in release mode. We were not alone in this. Other developers reported the same crash, for example this one and this one. You can enable the interpreter, but that has a huge performance cost, and it broke some other parts of the app.

So we went back to sqlite-net-pcl. But it still loads whole tables into memory and joins them in C#, not in SQLite. Joining the roads to their regions took over a minute for the administrators.

Don't get me wrong, there are ways to fix this, and I do not mean raw SQL, because that defeats the point of an ORM. I instead started asking myself a different question:

Why did we deal with any of this in the first place?

This all happened in the middle of 2024. That is when I started building SQLite.Framework on the side as a hobby, the main idea at the time was to have something similar to sqlite-net-pcl, but with full IQueryable support and no dynamic code generation.

Half a year later I had a working solution and released it on NuGet, so other people could give their feedback.

I decided to let the framework run as little code as possible and let the database handle the heavy lifting. I made it resilient enough that it even works with PublishAot, while also supporting features like JSON, JSONB, FTS5, R-Tree, window functions and more.

The work paid off. With SQLite.Framework, the login that once took over 30 minutes now takes about 3 minutes. And 2 of those 3 minutes are just pulling the data, not the database work.

30 min login with the old setup
3 min login with SQLite.Framework

And it is not just this app. In our benchmarks, bulk inserts run about 16 times faster than EF Core (on a computer without using AOT). See the benchmarks, or run them yourself.

Who is it for? Offline mobile apps, MAUI and Avalonia, and anything you ship with Native AOT. If you recognize your own setup here, the migration guides show the move step by step, from EF Core and from sqlite-net-pcl.
